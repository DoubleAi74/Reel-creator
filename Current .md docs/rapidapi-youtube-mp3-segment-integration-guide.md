# RapidAPI “YouTube to MP3” Segment API — Integration Guide

> **For an AI coding agent**
>
> Target service: **YouTube to mp3** by `marcocollatina` on RapidAPI  
> Verified: **1 July 2026**  
> RapidAPI listing: <https://rapidapi.com/marcocollatina/api/youtube-to-mp315>  
> API playground: <https://rapidapi.com/marcocollatina/api/youtube-to-mp315/playground>
>
> This is a third-party API, not an official YouTube API. Use it only for content the user owns, is licensed to use, or is otherwise authorized to download.

---

## 1. Objective

Implement a server-side integration that:

1. Accepts a YouTube URL.
2. Accepts a segment start and end time in seconds.
3. Requests an MP3 conversion from the RapidAPI service.
4. Polls the asynchronous conversion job until completion.
5. Downloads or streams the resulting temporary MP3 URL.
6. Handles timeouts, provider errors, malformed responses, and API limits safely.

The documented API endpoints are:

- `POST /download` — start an asynchronous conversion.
- `GET /status/{id}` — check conversion status and obtain the download URL.
- `GET /title` — optionally retrieve a video's title.

The documented `/download` parameters include:

- `url` — required YouTube URL.
- `format` — use `mp3`.
- `startTime` — optional clip start in seconds.
- `endTime` — optional clip end in seconds.
- `quality` — optional audio quality.
- `callbackUrl` — optional webhook URL.
- `X-RapidAPI-User` — optional provider-specific value.

## 2. Important implementation constraints

### Keep the API key server-side

Never call RapidAPI directly from browser code. Doing so exposes `X-RapidAPI-Key`.

Use one of:

- a Next.js Route Handler;
- an Express/Fastify backend;
- a server action that cannot expose secrets;
- a queue worker for longer conversion jobs.

### Treat the provider as asynchronous

Do not expect `POST /download` to return the finished MP3. It should return a conversion/job identifier. Use that identifier with `GET /status/{id}`.

### Do not hard-code an assumed response schema without testing

The public listing confirms the endpoints and parameters, but the exact live response fields are not reliably exposed outside the RapidAPI playground.

Before finalizing parsing logic:

1. Subscribe to the API.
2. Open the RapidAPI playground.
3. Run one short, authorized test conversion.
4. Save the exact JSON returned by both `/download` and `/status/{id}`.
5. Update the response-normalization functions and fixtures in this guide to match the live schema.

The implementation below intentionally supports several common field names.

## 6. Recommended server-side TypeScript client

Create:

```text
lib/youtube-mp3-provider.ts
```

```ts
const RAPIDAPI_HOST =
  process.env.RAPIDAPI_YOUTUBE_MP3_HOST ??
  "youtube-to-mp315.p.rapidapi.com";

const RAPIDAPI_KEY = process.env.RAPIDAPI_YOUTUBE_MP3_KEY;

const API_BASE_URL = `https://${RAPIDAPI_HOST}`;

export type StartConversionInput = {
  url: string;
  startTime: number;
  endTime: number;
  quality?: number;
};

export type ConversionState =
  | "queued"
  | "processing"
  | "complete"
  | "failed"
  | "unknown";

export type NormalizedConversionStatus = {
  state: ConversionState;
  downloadUrl?: string;
  message?: string;
  raw: unknown;
};

function requireApiKey(): string {
  if (!RAPIDAPI_KEY) {
    throw new Error("Missing RAPIDAPI_YOUTUBE_MP3_KEY");
  }

  return RAPIDAPI_KEY;
}

function headers(includeJson = false): HeadersInit {
  return {
    ...(includeJson ? { "Content-Type": "application/json" } : {}),
    "X-RapidAPI-Key": requireApiKey(),
    "X-RapidAPI-Host": RAPIDAPI_HOST,
  };
}

async function readProviderResponse(response: Response): Promise<unknown> {
  const text = await response.text();

  let body: unknown;

  try {
    body = text ? JSON.parse(text) : null;
  } catch {
    body = { unparsedBody: text };
  }

  if (!response.ok) {
    const error = new Error(
      `YouTube MP3 provider returned HTTP ${response.status}`,
    );

    Object.assign(error, {
      status: response.status,
      providerBody: body,
    });

    throw error;
  }

  return body;
}

function findString(
  value: unknown,
  keys: string[],
): string | undefined {
  if (!value || typeof value !== "object") {
    return undefined;
  }

  const record = value as Record<string, unknown>;

  for (const key of keys) {
    const candidate = record[key];

    if (typeof candidate === "string" && candidate.trim()) {
      return candidate;
    }
  }

  return undefined;
}

export async function startYoutubeMp3Conversion(
  input: StartConversionInput,
): Promise<{ id: string; raw: unknown }> {
  const response = await fetch(`${API_BASE_URL}/download`, {
    method: "POST",
    headers: headers(true),
    body: JSON.stringify({
      url: input.url,
      format: "mp3",
      startTime: input.startTime,
      endTime: input.endTime,
      ...(input.quality === undefined
        ? {}
        : { quality: input.quality }),
    }),
    cache: "no-store",
  });

  const raw = await readProviderResponse(response);

  const id = findString(raw, [
    "id",
    "jobId",
    "job_id",
    "conversionId",
    "conversion_id",
    "taskId",
    "task_id",
  ]);

  if (!id) {
    throw new Error(
      "Provider accepted the conversion but returned no recognizable job ID. " +
        "Inspect the live /download response and update the parser.",
    );
  }

  return { id, raw };
}

export async function getYoutubeMp3ConversionStatus(
  id: string,
): Promise<NormalizedConversionStatus> {
  const response = await fetch(
    `${API_BASE_URL}/status/${encodeURIComponent(id)}`,
    {
      method: "GET",
      headers: headers(false),
      cache: "no-store",
    },
  );

  const raw = await readProviderResponse(response);

  const statusText = (
    findString(raw, [
      "status",
      "state",
      "phase",
      "conversionStatus",
      "conversion_status",
    ]) ?? ""
  ).toLowerCase();

  const downloadUrl = findString(raw, [
    "url",
    "downloadUrl",
    "download_url",
    "fileUrl",
    "file_url",
    "link",
  ]);

  const message = findString(raw, [
    "message",
    "error",
    "detail",
    "reason",
  ]);

  if (
    downloadUrl ||
    ["done", "completed", "complete", "success", "finished"].includes(
      statusText,
    )
  ) {
    return {
      state: "complete",
      downloadUrl,
      message,
      raw,
    };
  }

  if (
    ["failed", "error", "cancelled", "canceled"].includes(statusText)
  ) {
    return {
      state: "failed",
      message,
      raw,
    };
  }

  if (
    ["queued", "pending", "waiting", "created"].includes(statusText)
  ) {
    return {
      state: "queued",
      message,
      raw,
    };
  }

  if (
    ["processing", "running", "converting", "downloading"].includes(
      statusText,
    )
  ) {
    return {
      state: "processing",
      message,
      raw,
    };
  }

  return {
    state: "unknown",
    downloadUrl,
    message,
    raw,
  };
}
```

After the first live API test, replace broad fallback parsing with a narrow schema validator based on the actual response.


Then define exact Zod schemas for the live `/download` and `/status` responses.

## 7. Input validation

Create strict validation before calling the provider.

```ts
import { z } from "zod";

export const youtubeMp3SegmentSchema = z
  .object({
    url: z
      .string()
      .url()
      .refine((value) => {
        const hostname = new URL(value).hostname.toLowerCase();

        return (
          hostname === "youtube.com" ||
          hostname.endsWith(".youtube.com") ||
          hostname === "youtu.be"
        );
      }, "A valid YouTube URL is required"),
    startTime: z.number().finite().min(0),
    endTime: z.number().finite().positive(),
    quality: z.number().int().positive().optional(),
  })
  .superRefine((value, context) => {
    if (value.endTime <= value.startTime) {
      context.addIssue({
        code: z.ZodIssueCode.custom,
        path: ["endTime"],
        message: "endTime must be greater than startTime",
      });
    }

    const duration = value.endTime - value.startTime;

    if (duration > 15 * 60) {
      context.addIssue({
        code: z.ZodIssueCode.custom,
        path: ["endTime"],
        message: "Requested segment exceeds the 15-minute application limit",
      });
    }
  });
```

Application limits should be configurable and should normally be stricter than provider limits.

Suggested initial limits:

```text
Minimum segment duration: 1 second
Maximum segment duration: 15 minutes
Maximum source video duration: determine after provider testing
Maximum polling duration: 3 minutes
Maximum concurrent conversions per user: 1–3
```

Also consider blocking playlists unless intentionally supported.

## 8. Recommended application API design

Do not keep one HTTP request open while repeatedly polling RapidAPI. Expose a job-based interface.

### Start conversion

```http
POST /api/youtube-audio-segments
Content-Type: application/json
```

Request:

```json
{
  "url": "https://www.youtube.com/watch?v=...",
  "startTime": 30,
  "endTime": 45
}
```

Response:

```json
{
  "jobId": "internal-job-id",
  "status": "queued"
}
```

### Read conversion status

```http
GET /api/youtube-audio-segments/{internalJobId}
```

Processing response:

```json
{
  "jobId": "internal-job-id",
  "status": "processing"
}
```

Complete response:

```json
{
  "jobId": "internal-job-id",
  "status": "complete",
  "downloadUrl": "/api/youtube-audio-segments/internal-job-id/file"
}
```

Use an **internal job ID**, not the provider's ID, in public URLs. Store the provider ID server-side.

## 9. Polling strategy

Suggested defaults:

```text
Initial delay: 1 second
Subsequent interval: 2–3 seconds
Maximum attempts: 60
Maximum elapsed time: 180 seconds
```

Use jitter to prevent synchronized polling:

```ts
const delayMs = 2000 + Math.floor(Math.random() * 500);
```

Stop polling when:

- the conversion completes;
- the provider reports failure;
- the maximum elapsed time is reached;
- the user cancels;
- the internal job expires.

Do not poll every few hundred milliseconds. Each status call may count toward the RapidAPI quota.

For scalable production use, perform provider polling in a background worker or durable job system rather than in the user's browser.

Possible tools:

- BullMQ with Redis;
- Inngest;
- Trigger.dev;
- Cloud Tasks;
- a database-backed worker;
- the application's existing background-job system.

## 10. Temporary download URLs

The API listing states that generated links are available for a maximum of approximately **one hour**.

Therefore:

1. Do not store the provider URL as a permanent asset.
2. Download the MP3 to controlled object storage immediately after completion.
3. Validate response size and content type.
4. Serve the stored object through your own authenticated endpoint or signed URL.
5. Delete it according to the application's retention policy.

Example server-side fetch:

```ts
export async function fetchCompletedMp3(
  providerUrl: string,
): Promise<Buffer> {
  const response = await fetch(providerUrl, {
    redirect: "follow",
    cache: "no-store",
  });

  if (!response.ok) {
    throw new Error(
      `Failed to fetch generated MP3: HTTP ${response.status}`,
    );
  }

  const contentType = response.headers.get("content-type") ?? "";

  const allowed =
    contentType.includes("audio/") ||
    contentType.includes("application/octet-stream");

  if (!allowed) {
    throw new Error(
      `Unexpected generated-file content type: ${contentType}`,
    );
  }

  const declaredLength = Number(
    response.headers.get("content-length") ?? "0",
  );

  const maximumBytes = 100 * 1024 * 1024;

  if (declaredLength > maximumBytes) {
    throw new Error("Generated file exceeds maximum allowed size");
  }

  const arrayBuffer = await response.arrayBuffer();

  if (arrayBuffer.byteLength > maximumBytes) {
    throw new Error("Generated file exceeds maximum allowed size");
  }

  return Buffer.from(arrayBuffer);
}
```

For larger files, stream to object storage rather than buffering the whole file in memory.

## 11. Security requirements

Implement all of the following:

- Keep the RapidAPI key server-only.
- Require authentication for conversion requests.
- Apply per-user and per-IP rate limits.
- Cap segment duration.
- Cap concurrent jobs.
- Validate that the URL belongs to YouTube.
- Do not accept arbitrary callback URLs from untrusted clients.
- Avoid logging API keys or complete provider responses.
- Redact temporary signed URLs in logs.
- Store provider IDs privately.
- Validate downloaded content type and size.
- Set timeouts on all provider requests.
- Prevent server-side request forgery when fetching result URLs.

### SSRF protection

The result URL is supplied by a third-party provider. Before server-side downloading, ideally restrict it to hostnames observed and approved during integration testing.

Do not build a general endpoint that fetches any URL supplied by a client.

## 12. Failure handling

Map provider failures into stable application errors.

Suggested error codes:

```text
INVALID_INPUT
UNSUPPORTED_VIDEO
SEGMENT_TOO_LONG
PROVIDER_RATE_LIMITED
PROVIDER_AUTH_FAILED
PROVIDER_REJECTED
PROVIDER_TIMEOUT
PROVIDER_MALFORMED_RESPONSE
CONVERSION_FAILED
RESULT_EXPIRED
INTERNAL_ERROR
```

Recommended HTTP handling:

| Situation | Application response |
|---|---|
| Invalid URL or times | `400` |
| User not authenticated | `401` |
| User quota exhausted | `429` |
| Provider returns `401` or `403` | `502`, log configuration error |
| Provider returns `429` | `503` or `429` with retry guidance |
| Provider returns `5xx` | `502` |
| Conversion timeout | `504` or terminal failed job |
| Unsupported/private/age-restricted video | `422` |
| Missing or expired result | `410` |

Do not expose the RapidAPI key, raw stack traces, or sensitive provider response fields to clients.

## 13. Timeouts and retries

Use explicit request timeouts:

```ts
const response = await fetch(url, {
  signal: AbortSignal.timeout(20_000),
  // other options
});
```

Retry only appropriate failures:

- retry network failures;
- retry `408`, `429`, and selected `5xx` responses;
- do not retry `400`, `401`, or `403` blindly;
- do not submit duplicate conversion jobs without idempotency protection.

Use exponential backoff:

```text
Attempt 1: immediate
Attempt 2: 1 second
Attempt 3: 2 seconds
Attempt 4: 4 seconds
```

Limit retries to avoid duplicate charges.

Create a deterministic request fingerprint from:


Before starting a provider conversion:

1. Search for a matching active or recently completed internal job.
2. Reuse it when appropriate.

## 15. Verification tests

The coding agent must add tests for:

### Validation

- valid standard YouTube URL;
- valid `youtu.be` URL;
- negative `startTime`;
- `endTime <= startTime`;
- segment longer than the application maximum;
- non-YouTube URL;
- malformed URL.

### Provider client

- extracts ID from the real start-response fixture;
- recognizes queued state;
- recognizes processing state;
- recognizes completion and extracts download URL;
- recognizes provider failure;
- rejects malformed success responses;
- maps `429`;
- maps `5xx`;
- times out correctly.

### Job flow

- starts exactly one provider job;
- avoids duplicates;
- polls until completion;
- stops on failure;
- stops on timeout;
- downloads the result;
- rejects an invalid result content type;
- rejects oversized results;
- never returns the RapidAPI key to the client.

## 19. Final caution

This integration depends on a small third-party provider whose interface and availability may change. Isolate it behind a provider adapter so it can be replaced without changing the rest of the application.

Recommended abstraction:

```ts
interface AudioSegmentProvider {
  start(input: StartConversionInput): Promise<{ providerJobId: string }>;
  status(providerJobId: string): Promise<NormalizedConversionStatus>;
}
```

Do not couple application routes, UI state, or database schemas directly to RapidAPI-specific response fields.
